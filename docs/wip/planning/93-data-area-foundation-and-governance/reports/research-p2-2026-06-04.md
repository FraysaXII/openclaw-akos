## 0. Load-bearing finding

Holistika already operates a **data-contract-shaped machine** without ever naming it: the
two-plane model (git-canonical CSV as schema SSOT + Supabase mirror as operational projection)
is a contract between a producer (the canonical) and consumers (mirror/FDW/graph readers). P2's
job is **not to invent contracts** — it is to **name the contract that already exists**, align
its vocabulary to the industry standard (ODCS v3.1), and make producer→consumer obligations
**declared and checkable** instead of implicit. This keeps the mint a *refinement of existing
Holistika practice* on the mechanics, and a *novel framing* only where it adopts ODCS/DAMA
vocabulary — so external citation is mandatory for the standard, light for the registry shape.

## 2. External research sweep

### 2.1 ODCS — Open Data Contract Standard v3.1.0 (Bitol / Linux Foundation, Dec 2025)

The industry standard for data contracts; Apache-2.0; strict-JSON-Schema as of v3.1
([bitol.io announcement](https://bitol.io/bitol-announces-odcs-v3-1-0-stronger-smarter-and-stricter/)).
Load-bearing structures we adopt the vocabulary of:

- **`schema`** — logical/physical data shape (the "what").
- **`quality`** rules — `library` (predefined metrics: `rowCount`, `nullValues`, `duplicateValues`,
  `invalidValues`, `missingValues`), `sql`, `custom` (Soda/Great Expectations/dbt). v3.1 adds
  `schedule`+`scheduler` → **executable**, not documentation
  ([data-quality.md](https://github.com/bitol-io/open-data-contract-standard/blob/main/docs/data-quality.md)).
- **`slaProperties`** — `property` / `value` / `unit` / `element` / **`driver`** ∈
  {`regulatory`, `analytics`, `operational`}; properties from the Data QoS table
  (`availability`, `retention`, `frequency`, `latency`, `timeToDetect`, `timeToNotify`,
  `timeToRepair`, `errorRate`)
  ([service-level-agreement.md](https://github.com/bitol-io/open-data-contract-standard/blob/main/docs/service-level-agreement.md)).
- **`team`** (owner + members + roles), **`version`**, **`status`**.

**Adoption posture:** align the **standard's vocabulary + the registry's columns** to ODCS so
contracts are portable and a future export to ODCS YAML is mechanical. We do **not** adopt the
full YAML toolchain at P2 (warn-first-fail-later); the lean CSV registry is the on-ramp.

### 2.2 DAMA-DMBOK Ch.3 — Data Governance function (DAMA International, 2017)

Data Governance = *"the exercise of authority, control, and shared decision-making (planning,
monitoring, enforcement) over the management of data assets"*
([DMBOK Ch.3 summary](https://www.scribd.com/document/988150549/Chapter-3)). Load-bearing for the policy:

- **Governance decides; management executes** — our policy sets the bar; Tech/DevOPS execute
  (mirror emit, DDL). This matches the DataOps "Tech implements, Data governs" split (`D-IH-93-C`).
- **Decision rights + accountability** — the policy must name *who decides what* (Data Owner =
  business-accountable; Data Steward = canonical hygiene; Data Governance Office = policy/contracts).
- **Operating models: centralized / decentralized / federated** — we pick **federated**
  ([Snowflake](https://www.snowflake.com/en/data-governance/models/),
  [Alation](https://www.alation.com/blog/understand-data-governance-models-centralized-decentralized-federated/)):
  central standards + domain execution. This is the charter's already-declared posture.

### 2.3 Data Mesh — federated computational governance (Dehghani, 2019)

Four principles ([martinfowler.com](https://martinfowler.com/articles/data-mesh-principles.html)):
domain-oriented ownership · **data as a product** · self-serve platform · **federated
computational governance** (global interoperability standards *baked computationally into the
platform*). Load-bearing: contracts are the **computational** form of governance — a CSV +
validator + mirror is exactly "policy baked into the platform," not a PDF policy. Each DATA-FAM
family (P6) is a **data product**; its contract is the product's published interface.

### 2.4 EDM Council DCAM (complement, not replacement)

DCAM (Data Capability Assessment Model) is the other industry governance framework cited
alongside DAMA ([Atlan](https://atlan.com/know/data-governance/data-governance-models/)). We
note it as the **maturity-assessment** lens (pairs with our area-completeness `compose_AREA`
bar) and DAMA as the **knowledge-area** lens. No conflict; cite DAMA as primary.

## 3. Synthesis → P2 design (Composer spec)

### 3.1 `DATA_GOVERNANCE_POLICY.md` — components (DAMA-grounded)

1. **Scope + definition** (DAMA: authority/control/shared decision-making over data assets).
2. **Governance vs management** boundary (decides vs executes; Data governs, Tech implements).
3. **Decision-rights matrix** — per decision class (schema change, classification, retention,
   contract approval) name the accountable role from `baseline_organisation.csv`.
4. **Federated operating model** — central standards (this policy + contract standard) + domain
   execution (areas own their data products); explicitly cite the data-mesh federated principle.
5. **Data Owner = business-accountable** (DAMA) vs Data Steward (hygiene) vs Data Governance Office.
6. **Enforcement = computational** (validators + mirror emit + drift gates), not document review.

### 3.2 `DATA_CONTRACT_STANDARD.md` — ODCS-aligned

Adopt ODCS v3.1 vocabulary: `schema` + `quality` (mapped to DATA-01..07) + `slaProperties`
(driver ∈ regulatory/analytics/operational) + `team`/ownership + `version` + `status`.
Posture: **contracts-as-code, warn-first-fail-later**; the CSV registry is the lean on-ramp,
ODCS YAML export is forward-chartered.
