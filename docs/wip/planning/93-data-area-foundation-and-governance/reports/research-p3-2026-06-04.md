# I93 P3 — research packet: three-tier data architecture + ODCS export

> Second architecture doctrine under the **§9.0 per-phase research bar**. Grounds
> `DATA_ARCHITECTURE.md` (git CSV ↔ Supabase ↔ Neo4j) and the **A++ ODCS export**
> script ratified at P2b. Composer reads this packet; it does not re-derive scope.

## 0. Load-bearing finding

Holistika **already runs a three-tier data architecture** — it was never named as one
plane:

| Tier | Store | SSOT role | Precedent |
|:---|:---|:---|:---|
| **T1 — Git canonical** | HLK v3.0 vault CSVs + Pydantic | Schema + governance SSOT | `PRECEDENCE.md` conflict resolution |
| **T2 — Supabase relational** | `compliance.*_mirror`, FDW, ops tables | Operational projection + read plane | `akos-holistika-operations.mdc` two-plane model |
| **T3 — Neo4j graph** | Projected nodes/edges | Rebuildable read index (discovery, lineage) | `akos/hlk_graph_model.py`; I07 closed |

P3's job is **name the tiers**, declare **per-canonical coverage** (which tiers each
register occupies), consume **I91 store inventory** without double-minting I91 P2
matrix work, and deliver **ODCS 3.1 YAML export** from `DATA_CONTRACT_REGISTRY.csv`
(DAMA L3 tool projection — operator A++ ratification).

## 2. External research sweep

### 2.1 DAMA-DMBOK Ch.2 — Data Architecture

DAMA separates **conceptual** (business entities), **logical** (technology-neutral
structures), and **physical** (implementation) views. Holistika mapping:

- **Conceptual** — process_list + engagement models + DATA-FAM umbrellas (P6).
- **Logical** — Pydantic field contracts + ODCS contract semantics.
- **Physical** — three tiers (git CSV, Supabase DDL/mirrors, Neo4j projection).

We adopt the **layer names** without importing medallion/lakehouse language unless P5
BI decision requires it.

### 2.2 Data mesh polyglot persistence (Dehghani 2019)

Domain data products may span multiple stores; **federated computational governance**
requires declared interfaces (contracts) across stores. Our `DATA_CONTRACT_REGISTRY`
rows per `(producer × data_surface)` implement this — P3 architecture doc makes the
store topology explicit.

### 2.3 OpenMetadata ODCS git-first workflow

OpenMetadata issue #21078 + ODCS API docs: contracts stored under source control,
imported via REST — **git remains authoritative**. Validates operator A++ posture:
export script produces ODCS YAML for catalog projection; no catalog-as-SSOT.
