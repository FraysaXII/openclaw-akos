## Why the sibling-repo carry-over is BLOCKED FROM AKOS workspace (D-IH-AGENT-EXEC scope statement)

[`.cursor/rules/akos-agent-checkpoint-discipline.mdc`](../../../../.cursor/rules/akos-agent-checkpoint-discipline.mdc) is the governing rule. The AKOS workspace at `c:\Users\Shadow\cd_shadow\openclaw-akos\` is the AKOS HLK SSOT root. Sibling repos `boilerplate`, `hlk-erp`, `kirbe-platform` live as separate `FraysaXII/<slug>` GitHub repositories with separate working trees the AKOS agent has no access to from this session. The plan's P2(d) deliverable explicitly is "PR on `FraysaXII/boilerplate` adding `boilerplate/playwright.config.ts`" — opening such a PR requires (a) a local sibling-repo checkout (operator initiates), (b) `gh pr create` against the right repository (operator-authenticated), (c) a smoke run of `npm run e2e` against the Vercel preview (operator-validated). All three are operator follow-up actions; the AKOS-side prerequisite (canonical template, validator, tests, wiring) is now in place so the operator can run a single command per sibling repo to copy the template and open the PR.

The agent **does not invent sibling-repo PRs from thin air** to honour the user's "do not stop until all to-dos are done" directive — doing so would violate [`.cursor/rules/akos-mirror-template.mdc`](../../../../.cursor/rules/akos-mirror-template.mdc) (which makes the AKOS template the SSOT and sibling repos the mirror, never the other way around) and [`.cursor/rules/akos-governance-remediation.mdc`](../../../../.cursor/rules/akos-governance-remediation.mdc) (which forbids creating files outside the AKOS root without explicit operator approval). Instead, the agent ships the AKOS-side P2 commit, files this pause record, and continues to AKOS-side work in subsequent phases that does NOT depend on sibling-repo working tree access (P4 validators, P5 SOP draft + workflow templates, P7 page-spec doc, P8 `akos-deploy-health.mdc` extension + I69 candidate scaffold).
