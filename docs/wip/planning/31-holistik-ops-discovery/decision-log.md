# Initiative 31 — Decision log

| ID | Date | Decision | Rationale | Status |
|:---|:-----|:---------|:----------|:-------|
| **D-IH-31-A** | 2026-04-30 | **Canonical default = audience of the artifact, not blanket EN.** Code, schemas, validators, registries, planning artifacts, brand-rule SSOT → English. Strategy artifacts whose deck-bound block feeds a single-locale external surface → audience language. External deliverables → audience language with locale-derived siblings. | A blanket EN-canonical rule would force a brand-voice rewrite on every change of an ES-bound strategy artifact (because BRAND_SPANISH_PATTERNS is not a translation discipline, it is a rewrite discipline). Audience-canonical avoids the round-trip cost. The cost is two flavours of canonical (EN + ES today; FR when needed); the gain is artifacts that ship in their target voice from day one. | **Approved** |
| **D-IH-31-B** | 2026-04-30 | **Persona registry tracks archetypes, not named individuals.** GOI/POI register stays the canonical for named individuals; PERSONA_REGISTRY records the shape of who's interacting (intent, value-band, typical channels, typical distance, qualification gate). | Two distinct concerns: archetype (the routing rule) vs identity (the specific person). Mixing them collapses the routing logic into an identity database, which would explode in size. Keeping them separate lets the persona registry stay short (~16 rows) and the GOI/POI register stay tactical. | **Approved** |
| **D-IH-31-C** | 2026-04-30 | **Channel touchpoint registry is distinct from CHANNEL_STRATEGY.md.** Touchpoint registry = where humans physically reach Holistika (LinkedIn DM, email, web form, Cal scheduling). CHANNEL_STRATEGY = acquisition hypothesis cards (founder-led outbound, partner B2B, KiRBe inbound, etc.). They cross-reference: each acquisition channel maps to one or more touchpoint channels. | Acquisition channels are about hypotheses for ICP × channel match (where do we go to find customers?). Touchpoint channels are about operational triage (when a human arrives, where exactly did they arrive?). The triage table is a different shape than the hypothesis table. | **Approved** |
| **D-IH-31-D** | 2026-04-30 | **Brand-voice rules per locale.** EN inherits BRAND_VOICE_FOUNDATION.md. ES uses BRAND_SPANISH_PATTERNS.md (already canonical). FR will use BRAND_FRENCH_PATTERNS.md, created here as a `status: stub` placeholder so it is discoverable when the first FR deliverable lands but is not yet a binding rule. | Stubs make the future obvious without committing to authoring work today. The first FR external deliverable will trigger an upgrade from stub to canonical. | **Approved** |
| **D-IH-31-E** | 2026-04-30 | **I30 strategy artifacts (MADEIRA_PLATFORM, GOVERNANCE_MOAT) stay in Spanish** under the new audience-canonical rule. They get `language: es` frontmatter; their deck-bound facts blocks already feed Spanish slides exclusively. | These are the textbook example of the audience-canonical exception (D-IH-31-A). Forcing them to English would create a translate-at-read-time cost for every deck rebuild. They stay Spanish; the SOP documents this is the right pattern. | **Approved** |
| **D-IH-31-F** | 2026-04-30 | **Outbound brief artifact format = Markdown with required sections** (Context, Deliverable, Quality gates, Brand voice rules, Timeline, Payment terms, Communication norms, Acceptance criteria). Operator can hand the rendered file to a non-Holistika contractor without explanation. | Markdown is human-readable and version-controlled. Required sections enforce that the brief is complete. The locale variants (EN/ES/FR) prove the locale-derivation pipeline from D-IH-31-A. | **Approved** |
| **D-IH-31-G** | 2026-04-30 | **Distance is N1-N4 ordinal** (terminal at N4; N5+ collapses) **+ `bridge_via` FK to another POI/GOI ID + `distance_assessed_date`**. Lives on GOI/POI register (named individuals — the operational source of truth) and as `typical_distance_band` on PERSONA_REGISTRY (archetype expectation). | Ordinal bands are queryable (`WHERE distance_band='N1'`); the bridge FK preserves the warm-intro chain for audit; the assessed date drives quarterly re-assessment cadence aligned with §6 service_role rotation. Storing the band on both archetype + identity surfaces lets the touchpoint kit pick a default template and the validator catch mismatches between expected (archetype) and actual (identity). | **Approved** |
| **D-IH-31-H** | 2026-04-30 | **Distance variants in the touchpoint kit live as sections within a single template file** (`## Variant — N1`, `## Variant — N2`, `## Variant — N3 / N4`), not as separate files. | Per-cell file count would explode to 8 cells × 4 distances × 1-3 languages = ~96 files for the seed pass alone. In-file variants keep the file count proportional (8-24 files for the seed) while preserving distance-routing precision. The brand voice constants stay in one place; only the salutation / bridge-reference / qualification-gate copy varies per N-band. | **Approved** |
