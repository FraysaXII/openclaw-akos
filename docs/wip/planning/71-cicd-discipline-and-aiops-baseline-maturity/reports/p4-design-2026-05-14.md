### 2.2 Why column-extension is the lower-friction default for mirrored canonicals

- **No new tables** to grant / RLS-policy-author / panel-wire / FK-resolve. Existing `compliance.*_mirror` tables already carry `service_role`-only RLS per the I59 / I14 inheritance.
- **No JOIN cost** at validator-time. The freshness check reads CSV columns directly (no Supabase round-trip; CSV is the SSOT for the validator).
- **Atomic per-row reads**. A `DECISION_REGISTER.csv` row carries its review stamp inline; no second-table lookup. ERP panel rendering (P5+ panel work) reads the same row.
- **Drift gate already covers it**. `scripts/validate_compliance_schema_drift.py` validates the CSV header ‚ÜĒ akos.* tuple alignment after the column extension ‚ÄĒ same gate as the I70 P8.5 baseline schema extension precedent.

The trade-off: if a future canonical is NOT mirrored (the Artifact class today), separate-table is the only path. P4 accepts the deferral.

## 3. Review-stamp shape ‚ÄĒ 4-column proposal (canonical)

Authored verbatim from `D-IH-71-E` proposal at I71 P0 charter:

| Column | Type | Nullable | Cross-link semantics | Notes |
|:---|:---|:---|:---|:---|
| `last_review_at` | `DATE` (Postgres) / ISO YYYY-MM-DD string in CSV | YES | none | When the operator last reviewed this row. Empty for newly-extended rows; populated incrementally per `OPERATOR_INBOX.md` workflow. |
| `last_review_by` | `TEXT` (Postgres) / free-form string in CSV | YES | FK-by-convention to `baseline_organisation.csv` `role_name` (advisory; not enforced at DB level) | Who reviewed. Plain `role_name` string (e.g. `PMO`, `System Owner`, `Brand Manager`). Free-form to allow operator-side roles outside the canonical list (e.g. `Founder` direct review). |
| `last_review_decision_id` | `TEXT` (Postgres) / free-form string in CSV | YES | FK-by-convention to `DECISION_REGISTER.csv` `decision_id` (advisory; validated by `validate_review_stamps.py` as `invalid-decision-ref` error if set but missing) | Optional `D-IH-*` decision row associated with the review. Set when the review produced a ratification; empty for routine "looked at it, still good" reviews. |
| `methodology_version_at_review` | `TEXT` (Postgres) / free-form string in CSV | YES | FK-by-convention to `LOGIC_CHANGE_LOG.md` methodology version section headers (advisory; future validator can cross-check) | Cross-link to `LOGIC_CHANGE_LOG.md` methodology version at time of review. Format: `vMAJOR.MINOR` per `D-IH-71-D` three-lane ratification (e.g. `v3.0`, `v3.1`). Empty for reviews that pre-date methodology-versioning awareness; populated for reviews 2026-05-14 onward by convention. |

**No CHECK constraints on column values.** Per lower-friction default + `D-IH-70-AC` precedent (additive CHECK constraints can land later via `NOT VALID + VALIDATE CONSTRAINT` if cardinality justifies). At P4 the columns ship as plain nullable TEXT/DATE with the validator script enforcing format-shape advisories.

**No NOT NULL constraints.** Empty values are the expected state for newly-extended rows (the operator backfills incrementally; the validator emits `missing-stamp` advisories rather than blocking).

## 4. SQL migration plan

### 4.1 DDL pattern per mirror table

```sql
ALTER TABLE compliance.<mirror_table_name>
  ADD COLUMN IF NOT EXISTS last_review_at DATE,
  ADD COLUMN IF NOT EXISTS last_review_by TEXT,
  ADD COLUMN IF NOT EXISTS last_review_decision_id TEXT,
  ADD COLUMN IF NOT EXISTS methodology_version_at_review TEXT;
```

Symmetric to I70 P8.2 baseline `ADD COLUMN IF NOT EXISTS` pattern (`20260513140000_i70_p82_baseline_sub_area_status.sql`). The `IF NOT EXISTS` idempotency guard makes the migration safe to re-apply after a partial failure (none expected; included as belt-and-suspenders).

**No `NOT VALID + VALIDATE CONSTRAINT`**: the I70 P8.5 precedent (`20260513150000_i70_p85_goipoi_stance_and_class_enum_extension.sql`) uses that pattern for `CHECK` constraints (enum widening). This P4 migration adds NULLABLE columns without CHECK constraints, so the simpler `ADD COLUMN IF NOT EXISTS` form suffices. The plan ¬ß"Risk register" R-71-3 mitigation ("`NOT VALID + VALIDATE CONSTRAINT` per I70 P8.5 precedent") applies to the **conceptual safety pattern** (additive-only, idempotent, no breaking change to existing rows) ‚ÄĒ which the simpler `ADD COLUMN` form satisfies for nullable additions.

### 4.2 Per-table migration sequence

In a single transaction:

1. `compliance.process_list_mirror` ‚ÄĒ 4 `ADD COLUMN IF NOT EXISTS`
2. `compliance.decision_register_mirror` ‚ÄĒ 4 `ADD COLUMN IF NOT EXISTS`
3. `compliance.initiative_registry_mirror` ‚ÄĒ 4 `ADD COLUMN IF NOT EXISTS`
4. `compliance.ops_register_mirror` ‚ÄĒ 4 `ADD COLUMN IF NOT EXISTS`

Total: 4 tables √ó 4 columns = **16 ALTER TABLE ADD COLUMN statements** in one transaction.

### 4.3 Backfill posture

**Empty by default.** Per kickoff Step 4: "Don't backfill row values yet ‚ÄĒ leave new columns blank (`""`) for all existing rows; the validator will surface them as 'missing-stamp' advisory hits, which operator backfills incrementally via `OPERATOR_INBOX.md` workflow."

The migration file includes an explicit `UPDATE ... SET last_review_at = COALESCE(last_review_at::DATE, NULL) WHERE last_review_at IS NULL;` no-op statement per the kickoff template ‚ÄĒ this is operationally a NULL-stays-NULL self-update that documents the backfill point. No real data migration runs at P4 because the columns are net-new (no pre-existing source column carries an authored / last_review-shaped value to project from).

The `INITIATIVE_REGISTRY.csv` carries a `last_review` column (YYYY-MM-DD) that is NOT the same semantic as `last_review_at` (the existing `last_review` is the operator's plan-update cadence; the new `last_review_at` is the doctrine-review cadence). Per lower-friction posture, P4 does NOT auto-populate `last_review_at` from `last_review` ‚ÄĒ they're orthogonal. The operator backfills `last_review_at` semantically per row.

### 4.4 Rollback plan

```sql
-- Symmetric DROP COLUMN per mirror table; same transaction:
ALTER TABLE compliance.process_list_mirror
  DROP COLUMN IF EXISTS last_review_at,
  DROP COLUMN IF EXISTS last_review_by,
  DROP COLUMN IF EXISTS last_review_decision_id,
  DROP COLUMN IF EXISTS methodology_version_at_review;
-- (repeat for decision_register_mirror, initiative_registry_mirror, ops_register_mirror)
```

Plus reverting CSV header rows + `akos.*` tuple updates via `git revert <P4-commit-sha>`.

If rollback is required, the operator runs the rollback DDL (provided verbatim in ¬ß6 of the SQL proposal doc) via `apply_migration` on the next free migration timestamp, then `git revert <P4-commit-sha>` to remove the CSV header columns + akos.* tuple entries + validator + tests + design/proposal/phase docs. The mirror tables return to their P3-state schema cleanly.

### 4.5 RLS posture

**No RLS policy changes.** All four mirror tables already carry `RLS = ON` with `service_role`-only deny-`anon`/-`authenticated` posture per the I14 / I59 mirror inheritance. The new columns inherit the same RLS by default (Postgres column-level RLS is not policy-scoped; RLS applies row-level to the whole row).

Per `.cursor/rules/akos-holistika-operations.mdc` ¬ß"Schema responsibilities": `compliance.*_mirror` carries `service_role`-only RLS for sync jobs. The new columns are populated via the same `scripts/sync_compliance_mirrors_from_csv.py` emit pattern (which inherits the SSOT tuple from `akos.hlk_*_csv.py`); no new write paths.

### 4.6 PII analysis

**None of the four new columns carry PII.**

- `last_review_at`: a date (operational metadata).
- `last_review_by`: a `role_name` string (organisational role; not a personal identifier ‚ÄĒ even if `Founder` resolves to a single individual, the SSOT is the org-internal role taxonomy in `baseline_organisation.csv`).
- `last_review_decision_id`: a governance-internal identifier (`D-IH-NN-X` pattern).
- `methodology_version_at_review`: a doctrine version string (`vMAJOR.MINOR`).

No GDPR / CCPA / sectoral-PII triggers. Aligns with the existing mirror-column posture (none of the existing process_list / decision_register / ops_register / initiative_registry columns carry PII either).

### 5.1 Per-CSV column position

The 4 new columns append at the **end** of each CSV header (after any trailing `notes` column). This positioning is deliberate:

- Append (vs insert) preserves all existing column positions, minimising blast radius for any downstream consumer that reads by index (none currently do; all read by name via `csv.DictReader` per the SSOT tuple, but the principle stays).
- After `notes` keeps `notes` as the operator's free-form trailing field; the review-stamp columns are governance-metadata that wraps around the trailing notes.

## 6. Validator extension ‚ÄĒ `scripts/validate_review_stamps.py`

New thin CLI authored alongside the migration. **Sibling to `validate_canonical_registry.py`**, not an extension of it (the existing canonical-registry validator scope is filesystem-presence + multi-claim detection; freshness-window is a different concern that gets its own script per the chassis-precedent pattern).

### 6.1 Logic (per kickoff Step 6)

Walk each canonical CSV in the registered set (4 CSVs at P4: process_list / decision_register / initiative_registry / ops_register). For each row:

| Rule | Severity | Logic |
|:---|:---|:---|
| `stale-row` | `warning` | `last_review_at` populated AND days-since-review > **180 days** (6-month threshold). |
| `missing-stamp` | `info` | `last_review_at` empty AND row's authored-date proxy (per CSV: `decided_at` for decisions; `inception_date` for initiatives; `opened_at` for ops; no proxy for process_list ‚ÄĒ flagged unconditionally for non-empty rows when stamp missing) > **30 days** ago. |
| `invalid-decision-ref` | `error` | `last_review_decision_id` set AND the ID does NOT exist in `DECISION_REGISTER.csv` `decision_id` column. |

### 6.2 CLI surface

```text
py scripts/validate_review_stamps.py
py scripts/validate_review_stamps.py --json-log
py scripts/validate_review_stamps.py --threshold-days 90
py scripts/validate_review_stamps.py --strict   # exit 1 on any warning OR error (default: exit 1 only on error)
```

### 6.3 Inbox surfacing

Stale + missing rows append a dated section to the sidecar `docs/wip/planning/REVIEW_STAMP_INBOX.md` (NEW file at P4; companion to `OPERATOR_INBOX.md` which is auto-rendered from `OPS_REGISTER.csv`). Idempotent: each run replaces the dated section block (markers `<!-- BEGIN REVIEW-STAMP-AUTO YYYY-MM-DD -->` / `<!-- END REVIEW-STAMP-AUTO -->`). Operator backfills review stamps in the canonical CSVs; subsequent runs drop those rows from the inbox.

### 6.4 Release-gate wiring

New row appended to `scripts/release-gate.py` adjacent to the existing operator-inbox / freshness-canary INFO rows:

```text
[INFO] Review-stamp freshness (scripts/validate_review_stamps.py ‚ÄĒ 4 mirrored canonicals; 180-day window; advisory only)
```

Severity: `INFO` (non-blocking). Per the kickoff "Wire into `scripts/release-gate.py` as a new advisory row (INFO level; doesn't block)" ‚ÄĒ the freshness signal is operator-actionable, not CI-blocking. R-71-1 mitigation (validator false positives erode trust) applies: stale-row + missing-stamp emit warnings/info per the severity ladder; only `invalid-decision-ref` emits an `error` (which validate_review_stamps.py exits 1 on, but release-gate's `INFO` wrapper makes the row non-blocking ‚ÄĒ same posture as `Operator inbox stale` row).

## 9. Risk + rollback

| Risk | Severity | Mitigation |
|:---|:---:|:---|
| Migration partial-apply leaves some mirrors with new columns + others without | High | Single transaction (`BEGIN; ... COMMIT;`); `IF NOT EXISTS` idempotent guard. Rollback DDL provided verbatim in SQL proposal ¬ß6. |
| CSV header drift between the 4 extended canonicals + akos.* tuples | High | `validate_compliance_schema_drift.py` runs in release-gate and locks header ‚ÜĒ tuple alignment per the I70 P8.5 precedent. |
| validate_review_stamps false positives on newly-extended rows | Medium | `missing-stamp` is `info` severity (not `error`); release-gate wraps as INFO row (non-blocking); operator backfills incrementally. The 30-day grace from the row's authored-date proxy reduces noise on rows authored ‚Č§ 30 days. |
| Panel slot reservation collides with future I72+ RevOps panel naming | Low | Reserved path `/operator/governance/freshness-dashboard/` is generic enough to absorb either P4 stamp work OR future RevOps freshness work without rename. If RevOps wants a dedicated `/operator/revops/freshness-dashboard/` instead, this slot remains valid for the governance-trio mirrors and the new RevOps slot lands separately. |
| Backfill workflow drift (operator forgets to update REVIEW_STAMP_INBOX.md surfacing once backfill happens) | Low | Validator is idempotent; re-running drops backfilled rows from the inbox automatically. No operator workflow burden beyond editing CSV cells. |

### 9.1 Half-applied migration recovery

If the migration succeeds partially (e.g. 2 of 4 mirror tables get the columns, then a network blip aborts):

1. Re-run the same `apply_migration` call. `IF NOT EXISTS` guards make the already-applied ALTERs no-ops.
2. The remaining ALTERs apply on the second run.
3. `list_tables verbose=true` confirms all 4 mirror tables carry the 4 new columns.
4. If the second run also fails: STOP per opt-stop-report posture; author `reports/p4-blocker-2026-05-14.md` with the failure detail; the rollback DDL drops any partially-applied columns; then escalate.

The CSV header + akos.* tuple updates land via git regardless of migration state ‚ÄĒ they're decoupled from the live mirror until the next `sync_compliance_mirrors_from_csv.py` run.
