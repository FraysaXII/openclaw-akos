# v3.1 Schema Migration Log — `baseline_organisation.csv` extension (sub_area + status)

> **Why this doc exists.** Operator-stated rationale at the §8.14 inline-ratify gate: *"this kind of decisions will resurface a lot now that we're getting close to v3.1, because some design (no matter the area or process) were different or had a different intent from my side. That's why we call this methodology versionable. Thanks for detecting this and stay alert for more. This is a huge plan so checkpoints and carefully crafted documentation will be important."*
>
> This is the first **v3.1 methodology versioning checkpoint**. It captures the audit trail for the kind of schema/intent gap that will surface repeatedly as I70 closes and v3.1 lands. Future similar events follow this template.

## 1. The gap detected

When I70 P8 §8.14 (P8.2) reached the canonical-CSV migration step, the agent surfaced a mismatch between the plan and the live schema:

- **Plan-described schema** (§8.14, drafted at I70 inception): `baseline_organisation.csv` columns include `role_id, sub_area, parent_role, role_type, hourly_rate_eur, status, notes`.
- **Live canonical schema** (as of 2026-05-13): `org_uuid, role_name, role_description, role_full_description, access_level, reports_to, area, entity, org_id, sop_url, responsible_processes, components_used, role_hourly_min_eur, role_hourly_par_eur, role_hourly_max_eur` — **no `sub_area`, no `status`, no `parent_role`, no `role_type`, no `notes` columns**.
- **`process_list.csv`** uses `area=MKT` (3-letter) with hierarchical `thi_mkt_prj/ws/dtp_<n>` IDs, **not** the plan's `proc_mkt_reach_*` style.

This is a **plan/reality drift** that surfaced at execution time, not at planning time. Per `.cursor/rules/akos-inline-ratification.mdc`, the surface was an inline-ratify gate via `AskQuestion`, not a real-stop pause.

## 2. The decision (D-IH-70-Z)

**Schema extension (Option C ratified)** — extend `baseline_organisation.csv` with two new columns; do not backfit the plan to the legacy structure.

- **`sub_area`** (TEXT, nullable/empty): hierarchical encoding within an area. Examples: `Marketing/Reach`, `People/Compliance`, `Operations/SMO`. Empty for areas without sub-divisions.
- **`status`** (TEXT, CHECK-constrained: `active` / `deprecated` / `pending` / NULL): soft-state tracking for future deprecations and pending-ratification rows.

**Hard-remove (Option B ratified)** — for the Marketing M3 redesign legacy roles (`Social Media Manager`, `Growth Manager`), do **not** soft-deprecate. Delete the rows entirely; reassign orphan `process_list.csv` rows to new M3 sub-area roles.

Operator quote: *"Option B for this and any other. If i didn't do this for people im sorry but this is a v3.1 rework run and everything needs to be as of today."*

The hard-remove rule is therefore **the v3.1 default for I70-era legacy migrations**. Sister phases §8.15 (P8.3 People restructure: Talent monolith → 4 sub-roles) inherit this rule.

## 5. Rollback plan

If a future v3.x decides to revert this:

1. Drop the schema columns: `ALTER TABLE compliance.baseline_organisation_mirror DROP COLUMN sub_area, DROP COLUMN status; DROP INDEX compliance.baseline_organisation_mirror_area_sub_area_status_idx;` Reapply via a forward migration (no down-migration in this codebase per the `supabase/migrations/` convention).
2. Trim the FIELDNAMES tuple in `akos/hlk_baseline_org_csv.py` to its 15-column shape; trim `OrgRole` Pydantic fields accordingly.
3. Re-run `scripts/sync_compliance_mirrors_from_csv.py` to refresh the mirror; column-drift validator will fail until step 4.
4. Edit `baseline_organisation.csv` to remove the `sub_area` + `status` columns from header + every row.
5. Restoring the hard-removed `Growth Manager` + `Social Media Manager` rows would require a separate decision — re-create the rows from git history (`git show 63792a9:docs/references/hlk/v3.0/Admin/O5-1/People/Compliance/canonicals/baseline_organisation.csv`); their org_uuids are stable.
6. Re-assign `process_list.csv` orphans back; their pre-migration `(role_parent_1, role_owner)` pairs are recorded in this checkpoint §3.3. The migration scripts (`scripts/_p82_marketing_m3_*.py`) followed the W2/W3 temp-script lifecycle (deleted in this commit after successful execution). Full pre-migration row content is in the parent commit `git show <P8.1-sha>:docs/references/hlk/v3.0/Admin/O5-1/People/Compliance/canonicals/process_list.csv` and `baseline_organisation.csv`.

## 6. Pattern for future v3.1 schema/intent gaps

When the next plan-vs-reality drift surfaces (it will — operator confirmed): use this checkpoint as the template. The 6-step protocol:

1. **Detect** at execution time (agent grep + Read of plan vs canonical).
2. **Surface** via inline-AskQuestion (not a real-stop pause); offer 2-3 options with rationale.
3. **Ratify** operator decision; assign next `D-IH-NN-XX` ID.
4. **Execute** the schema+row+propagation triple in one atomic commit.
5. **Document** here under `docs/wip/planning/<initiative>/checkpoints/v3.x-schema-migration-log-<date>.md`.
6. **Cross-reference** the checkpoint from `DECISION_REGISTER.csv.notes` and the relevant phase plan section.

The checkpoint becomes the audit trail. Operator's "I'm here for you anytime" promise is reciprocated by **never losing the operator** in the diff: every schema decision lands with a full rollback plan + verification matrix + pattern-future hook.
