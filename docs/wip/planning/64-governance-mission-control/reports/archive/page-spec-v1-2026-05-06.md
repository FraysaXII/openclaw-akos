# Page spec — Governance Mission Control (hlk-erp)

**Route:** `/operator/governance/external-repos/`
**Auth:** operator-only (RBAC via existing hlk-erp middleware)
**i18n:** en + es
**Theme:** matches existing hlk-erp shadcn/ui pattern

## Layout (desktop, ≥1280px)

```
+------------------------------------------------------------------+
|  Governance Mission Control          [refresh]  [demo-mode toggle]
+------------------------------------------------------------------+
| Panel 1 (full width): Repo health grid                          |
+------------------------------------------------------------------+
| Panel 2 (1/2)          | Panel 3 (1/2)                          |
| Bless this repo        | Drift inbox                            |
+------------------------+----------------------------------------+
| Panel 4 (1/2)          | Panel 5 (1/2)                          |
| Secret rotation        | Canonical change broadcast log         |
+------------------------+----------------------------------------+
| Panel 6 (full width): SOP + decision-log surface                |
+------------------------------------------------------------------+
```

On mobile, panels stack vertically. Panels 2-5 collapse to
accordions; panel 1 becomes a horizontally scrollable card list.

## Panel 1 — Repo health grid

**Source:** `compliance.repository_registry_mirror` joined with
`compliance.repo_health_snapshot_mirror`.

**Columns (left to right):**

| Column | Source | Display |
|:---|:---|:---|
| repo_slug | registry | code-styled badge |
| class | registry | tag (platform / internal / client-delivery / reference) |
| blessed | registry.notes flag + snapshot.last_blessed_at | ✓ if `last_blessed_at` ≤ 30d, else ⚠ |
| drift | snapshot.akos_mirror_sha256_match | ✓ if true, ✗ if false |
| ci_posture | last `check_external_repo_ci_posture.py` run summary | percentage 0-100% |
| license | snapshot.license_present | ✓/✗ |
| sentry | last live check from `--auto-fix` history | ✓/⚠/✗ |
| last_commit_main | GitHub API (cached 5min) | relative time |
| secrets_due_30d | snapshot.secret_rotation_oldest_age_days vs 75d threshold | count |
| actions | clickable | "Open" → drill-down panel below |

**Color logic:**
- All ✓ → row background green-50
- One or more ⚠ → row background amber-50
- One or more ✗ → row background red-50

**Interactions:**
- Click row → expands a detail accordion below the grid showing the
  per-check breakdown and links to the underlying SHAs / runs.
- "Refresh" button at the top forces a re-fetch (bypasses TanStack
  Query cache).

**Empty state:** "No repos in REPOSITORY_REGISTRY.csv yet — bless your
first repo via `py scripts/bless_external_repo.py --repo-slug <name>`
in the AKOS workspace."

## Panel 2 — Bless this repo

**Source:** rows in `REPOSITORY_REGISTRY.csv` where the AKOS-side
`detect_unblessed_registry_rows.py` reports `BLESSED == false` (i.e.
the SHA-bless folder doesn't exist OR is older than 90 days).

**UI:**
- Picker: select a slug from the unblessed list.
- Class indicator (read-only).
- Optional `--with` features checkboxes (sentry, slack, postman, sbom,
  semgrep, codecov, semantic-release, browser-smoke).
- "Bless" button — opens a confirm modal: *"This will trigger a
  GitHub Actions workflow that opens a PR against `<slug>`. Continue?"*
- On confirm: dispatches `governance-bless-repo.yml` workflow.
- Live status: polls the workflow run; renders ✓ + PR URL when done.

**Empty state:** "Every repo in REPOSITORY_REGISTRY is blessed and
within freshness window. Add a new row in AKOS to register a new
consumer."

## Panel 3 — Drift inbox

**Source:** all rows where `snapshot.akos_mirror_sha256_match == false`.

**UI:**
- One card per drifting repo. Card shows:
  - Slug + class
  - Files that drifted (the SHA stamps under
    `.github/.akos-bless/*.sha256` that don't match AKOS canon)
  - "Open auto-PR" button — confirm modal → dispatches
    `governance-bless-repo.yml` with `auto_pr: true`.

**Empty state:** "No drift detected. All blessed repos match AKOS
canonical templates." (green checkmark)

## Panel 4 — Secret rotation calendar

**Source:** `lib/governance/secrets.ts` reads each blessed repo's
`docs/runbooks/secrets-rotation.md` frontmatter (or the
`.github/.akos-bless/secret_rotation_metadata.json` if the runbook
isn't populated yet).

**UI:**
- Heatmap (last 30 days + next 60 days). X-axis = day. Y-axis = repo.
  Cell color encodes urgency.
- Below the heatmap: a list of secrets with rotation due in <14 days,
  ranked by urgency.
- Each list row links to the secret's runbook section + the GitHub
  secrets settings URL.

**Empty state:** "No tracked secrets. Bless your repos with the
secrets-rotation.md runbook to enable this panel." (with a
deeplink to `secrets-walkthrough-2026-05-06.md`)

## Panel 5 — Canonical change broadcast log

**Source:** `governance.canonical_change_log` (Supabase table added
in I64 P1; populated by `notify_consumers_of_canonical_change.py`
when it runs in non-dry-run mode).

**Columns:**

| Column | Display |
|:---|:---|
| changed_at | relative time |
| csv_name | code badge (e.g. `POLICY_REGISTRY`) |
| consumers | list of slugs that consume this mirror |
| broadcast_status | per-consumer: posted / regen-PR / ack |
| actions | "Open Slack thread" link if a thread URL was captured |

**UI:** TanStack Table; default sort by `changed_at DESC`; pagination
20 per page; date range filter.

**Empty state:** "No canonical changes broadcast in the last 30 days."

## Panel 6 — SOP + decision-log surface

**Source:** static map (compiled at build time) from each automated
process to its SOP markdown + decision-log entry.

**UI:** 3 cards (one per process), each with:
- Process name
- Active SOP filename (link to GitHub source)
- `role_owner` (DevOPS or System Owner)
- Linked decisions (D-IH-63-D / D-IH-63-E / D-IH-63-F)
- Last review date

**No empty state** — these always render.

## Data contracts

**TypeScript interfaces** (generated by `regen_consumer_types.py`):

```ts
interface RepoHealthRow {
  repo_slug: string;
  github_url: string;
  class: 'platform' | 'internal' | 'client-delivery' | 'reference';
  blessed: boolean;
  last_blessed_at: string | null;
  drift_status: 'ok' | 'warn' | 'fail';
  ci_posture_pct: number;
  license_present: boolean;
  sentry_status: 'ok' | 'warn' | 'fail' | 'unknown';
  last_commit_main: string | null;
  secrets_due_30d: number;
}

interface DriftEntry {
  repo_slug: string;
  drifting_files: string[];
  detected_at: string;
}

interface CanonicalChangeLogEntry {
  id: string;
  changed_at: string;
  csv_name: string;
  consumers: { slug: string; status: 'posted' | 'regen-pr' | 'ack' }[];
  thread_url: string | null;
}
```

## Error states

| Scenario | UX |
|:---|:---|
| Supabase down | Each panel renders cached data + "stale by Xm" banner; refresh disabled |
| AKOS GitHub Actions workflow fails | Bless / drift action panels render the error from the workflow job summary; suggest manual `--dry-run` from CLI |
| Demo mode active | Panels render fixture data + "Demo mode" banner; write actions disabled |
| User loses operator role mid-session | Refresh forces re-auth; subsequent reads return 403 |

## Open questions for the operator

1. **Subdomain**: host under `showcase.holistikaresearch.com/operator/governance/...`
   or carve out a separate `governance.holistikaresearch.com`? Default plan
   is to share with MADEIRA to avoid auth fragmentation.
2. **Write actions in P3 or DEFERRED**: ship read-only first (P2),
   add write actions in P3, or defer write actions to a follow-up
   initiative? Current plan: P3 with feature flag
   `GOVERNANCE_WRITE_ACTIONS_ENABLED=false` in non-prod by default.
3. **Demo data**: should the demo mode use fixture repos (to avoid
   showing real repo names in screenshots / demos) or just inject
   "demo" badges on the real repos? Current plan: fixture repos
   under `kirbe-demo`, `hlk-erp-demo`, `boilerplate-demo` slugs.
