## §2 Executive calls made by the subagent

### §2.1 Integrity-matrix methodology — 5-signal model with semantic substring scan (closes D-IH-81-F)

**Executive call:** The vault-integrity matrix carries **5 coverage signals per executable row**: `knowledge_pairing_status` + `paired_sop_status` + `mirror_coverage_status` + `audience_tags_status` + `cadence_status`. Aggregated to a **3-value row verdict** (`pass` / `partial` / `fail`) via deterministic [`KbIntegrityMatrixRow.compute_verdict()`](../../../../akos/hlk_kb_integrity.py). Pass-rate threshold = **95%** per the existing operator practice on canonical gates (matches I71/I80 retrofit threshold cadence). Threshold is **not enforced** at P1 baseline (CI INFO advisory); promotion to strict mode is gated at I81 P9 closure UAT.

**Executable predicate**: `item_granularity in {"task", "process"}` (1085 rows in the real corpus today; the other granularities `project` (15) + `workstream` (70) are scope/structure aggregates that do not name discrete executable units).

**Signal-detection heuristics**:

| Signal | Heuristic |
|:---|:---|
| `knowledge_pairing_status` | Substring scan: item_id appears in any cell of `KNOWLEDGE_PAIRING_REGISTRY.csv` `pairing_id` / `parent_doc_path` / `companion_doc_paths`. |
| `paired_sop_status` | Substring scan: item_id appears in any `SOP-*.md` file under `docs/references/hlk/v3.0/**` (102 files scanned at this commit; 89 item_ids matched = 8.20%). |
| `mirror_coverage_status` | Defaults to `covered_by_emit` by construction of the `compliance_mirror_emit` sync flow; future commits can flip individual rows to `mirror_skip` when intentionally not mirrored. |
| `audience_tags_status` | Defaults to `deferred` for every row at P1 baseline; live wire forward-chartered to a successor I81 P1 follow-up commit that joins via I85 P1 `AUDIENCE_REGISTRY`. |
| `cadence_status` | Lookup: process_list `cadence_type` cell in `{on_demand, scheduled, event_triggered, gated_operator}` per [`akos-executable-process-catalog.mdc`](../../../../.cursor/rules/akos-executable-process-catalog.mdc) RULE 3. |

**Rationale:** Pragmatic at the chassis layer. Substring scans are honest (the operator can read each one) + cheap (KP registry has ~5-10 rows; SOP corpus has ~100 files). FK-style exact matching (e.g., SOP frontmatter `process_item_ids:` list) is forward-charterable when the SOP retrofit waves stabilize that shape. The 95% threshold matches operator precedent without inventing a new gate semantics.

**Reversibility:** Single-diff at three sites — (a) edit signal heuristics in [`scripts/audit_kb_integrity.py`](../../../../scripts/audit_kb_integrity.py) `_resolve_item_ids_in_pairing_rows` / `_scan_v3_sop_corpus_for_item_ids`; (b) tune threshold via `--pass-threshold` flag or default; (c) promote/demote `--strict` enforcement at I81 P9 closure or earlier per operator decision.

**Operator override mechanism:** Reply with a heuristic change request ("use FK-style `process_item_ids:` frontmatter scan instead of substring", or "promote threshold to 98%") and I'll author the follow-up commit; or silently accept (default if no override within 14 days per [`akos-agent-checkpoint-discipline.mdc`](../../../../.cursor/rules/akos-agent-checkpoint-discipline.mdc) soft-pause auto-clear).

### §2.2 Matrix asset classification — report-class, not canonical SSOT

**Executive call:** The matrix CSV + audit narrative under `reports/i81/` are **reference-class** (planning-internal), not canonical SSOT. They are NOT registered in [`PRECEDENCE.md`](../../../references/hlk/v3.0/Admin/O5-1/People/Compliance/canonicals/PRECEDENCE.md). The canonical SSOT for the data they aggregate stays in `process_list.csv` (source rows) + `KNOWLEDGE_PAIRING_REGISTRY.csv` (pairing rows) + the v3.0 SOP corpus (markdown files).

**Rationale:** Matches master-roadmap §6 asset classification ("Reference (planning-internal): `reports/i81/kb-integrity-audit-*.md`, `reports/i81/kb-integrity-matrix-*.csv`"). The matrix is a *derived view* — re-emittable any time by re-running `py scripts/audit_kb_integrity.py --date <YYYY-MM-DD>` from the canonical sources. Registering it as canonical would create dual-SSOT confusion (the audit script becomes the truth source instead of the underlying data).

**Reversibility:** Single-diff at [`PRECEDENCE.md`](../../../references/hlk/v3.0/Admin/O5-1/People/Compliance/canonicals/PRECEDENCE.md) — append a `compliance.kb_integrity_matrix_mirror` row if the operator wants the matrix mirrored to Supabase as a queryable table. Today no mirror exists; the audit script writes flat artifacts.

**Operator override mechanism:** Reply "register matrix as canonical + mint Supabase mirror" → I'll author the PRECEDENCE row + supabase/migrations entry per [`akos-holistika-operations.mdc`](../../../../.cursor/rules/akos-holistika-operations.mdc) "New git-canonical compliance registers (pattern)".

### §2.3 Audience-tags wire — deferred to follow-up commit

**Executive call:** The `audience_tags_status` signal is `deferred` for every row at P1 baseline. The live wire (joining `process_list.role_owner` → `baseline_organisation.csv` → I85 `AUDIENCE_REGISTRY.csv` audience tag) is forward-chartered to a successor I81 P1 follow-up commit. The matrix schema already carries the column with the `deferred` literal so the upgrade is a single-file change.

**Rationale:** Three convergent reasons. (1) The I85 `AUDIENCE_REGISTRY.csv` shipped at SHA `7d47199` (2026-05-16) is stable, but the role_owner → audience-tag join requires walking `baseline_organisation.csv` + the I79 People-DoD overlays to resolve roles consistently across mirrored canonicals — that's an own commit's worth of work. (2) The pass_rate metric is necessarily ~0% until this wire lands (every row pulled below `pass` by the deferred signal); shipping the matrix with the gap clearly labeled is more honest than artificially boosting the metric. (3) The follow-up commit can land any time; nothing in P2-P9 of I81 blocks on it.

**Reversibility:** Drop-in at [`scripts/audit_kb_integrity.py::build_matrix_rows`](../../../../scripts/audit_kb_integrity.py) — when the join logic lands, replace the hard-coded `audience_tags_status="deferred"` with a lookup against the AUDIENCE_REGISTRY join result. Pydantic enum already accepts `"matched"` / `"deferred"` (no schema change needed).

**Operator override mechanism:** When ready, the follow-up commit can be authored by either (a) the parent agent in the next chat turn, (b) a fresh subagent dispatched specifically for the wire, or (c) operator inline-ratify of a draft change list. No code lock-in.
