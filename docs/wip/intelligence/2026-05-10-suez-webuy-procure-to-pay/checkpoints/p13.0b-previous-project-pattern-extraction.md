# I70 P2 — Previous-project pattern extraction (annex)

> Read-only annex extracting reusable patterns from the operator's prior product-owner work on the Shopify Bundle + Cart App project (Holistika x Websitz x EFA, Q1-Q2 2026). Each of the 6 source files maps to a candidate `_engagement-template/01-operator-pack/_examples/` reference-only slot, so future engagements (especially type-3 SaaS-shape and type-2 partner-collaboration types) can be authored by template-fork rather than from-scratch. **Reference-only classification** per Phase 4 §11 — these patterns inform but never drive canonical SSOT.

## 1. Why this dig

The operator explicitly named `docs/references/hlk/previous-project-for-product-owner-example-only/` as prior art to mine for I70 (per F-70 "every word in my prompt is curated"). The folder contains 6 mature documents from the Shopify Bundle + Cart App engagement — each one represents a **shape pattern** Holistika has executed end-to-end and could reuse. The annex extracts the section-structure pattern from each + maps to engagement-template slots + flags any forward-looking governance candidate.

## 2. Six pattern extractions

### 2.1 `product-requirements.md` (PRD shape, ~18,500 chars)

**Section structure observed:**
1. Executive Summary (1 paragraph customer-facing thesis)
2. Partnership Roles (3-row table: party / representative / responsibilities)
3. Functional Requirements (numbered hierarchy: 3.1 cart drawer / 3.2 bundles / 3.3 analytics / etc.)
4. Non-Functional Requirements (performance / security / observability)
5. User Stories (per-persona acceptance criteria)
6. Out of Scope (explicit rejections)
7. Open Questions (operator review items)

**Maps to engagement-template slot:** `_engagement-template/01-operator-pack/_examples/PRD-template.md` (NEW reserved slot).

**Type-applicability:** Best fit for **type-3 product engagements** (SaaS shape; has clear product features + multiple counterparties + commercial timeline). Good partial fit for type-2 partner collaborations where a co-built product is the deliverable.

**Forward-looking note:** When the first I70 + I71 engagement materializes that needs a PRD shape (e.g., MADEIRA productization in Phase 4.8 trigger 1, or a future Holistika SaaS pilot), this template gets activated.

### 2.2 `go-to-market-strategy.md` (GTM shape, ~18,700 chars)

**Section structure observed:**
1. Strategic Foundation (1.1 Core Insight; 1.2 Target Audience; 1.3 Reference model — uses Lemlist's flywheel as analog)
2. Positioning (Holistika's brand promise vs competitor positioning)
3. Go-to-Market Phases (per-phase named: Soft Launch / Community Validation / Scaling / Expansion)
4. Channel Strategy (Owned / Earned / Paid breakdown)
5. Partnership Channel (Websitz + EFA roles)
6. Pricing Strategy (tiers + ICP alignment)
7. Success Metrics (per-phase KPIs)
8. Risks + Mitigations

**Maps to engagement-template slot:** `_engagement-template/01-operator-pack/_examples/GTM-template.md` (NEW reserved slot) + cross-reference from `Marketing/Reach/SOP-GTM_QUALIFICATION_001.md` (active per I66 P5 incr 9; this template extends with full launch-phase deliverable structure).

**Type-applicability:** Best fit for any engagement with a launch-phase deliverable (type-1 customer + type-3 product + type-2 partner). The Lemlist flywheel reference and the per-phase named structure are highly reusable.

**Forward-looking note:** I72 (Marketing area governance, renamed from I67) will likely promote this template to `Marketing/Reach/canonicals/GTM_PLAYBOOK.md` after federating it through Phase 4.5.

### 2.3 `technical-specification.md` (tech spec shape, ~25,200 chars)

**Section structure observed:**
1. Architecture Overview (3-layer split: Embedded App / Storefront Extensions / Backend)
2. Component Inventory (per-layer modules with technology choices)
3. Data Model (entities + relationships + diagram)
4. API Surface (endpoint catalog with request/response shapes)
5. Integrations (Shopify GraphQL Admin API / Functions / etc.)
6. Security Posture (auth / data / compliance)
7. Observability + Performance (metrics / SLOs / logging)
8. Build + Deploy (CI/CD pipeline)
9. Migration Strategy (Scripts sunset → Functions transition)
10. Open Technical Questions

**Maps to engagement-template slot:** `_engagement-template/01-operator-pack/_examples/TECH-SPEC-template.md` (NEW reserved slot). Cross-reference from [`SOP-CICD_BASELINE_001.md`](../../../references/hlk/v3.0/Admin/O5-1/Tech/System%20Owner/SOP-CICD_BASELINE_001.md) (I68 P5) — the CICD baseline section shape mirrors this tech spec's §8 Build + Deploy structure.

**Type-applicability:** Tech-heavy engagements (type-2 partner with co-built product, type-3 product). Maps cleanly to the SUEZ pattern: cdc-feasibility-shape.fr.md is essentially a tech-spec for the WeBuy automation, just in customer-facing register.

**Forward-looking note:** When I71 CICD + AI-ops baseline ships its operator-facing companion docs, this template's §6 + §7 + §8 structure will inform it.

### 2.4 `project-timeline.md` (timeline shape, ~15,600 chars)

**Section structure observed:**
1. Timeline Summary (single phase table with named phases + duration + dates + key deliverable)
2. Per-phase deep-section (Phase 0 / Phase 1 / etc.) with: Goal / Deliverables / Acceptance Criteria / Risks / Owner
3. External constraints (Shopify Scripts sunset June 30, 2026 — load-bearing dependency)
4. Resource allocation (per-phase staffing + budget)
5. Critical path (Mermaid Gantt or table)
6. Decision points (per-phase operator gates)

**Maps to engagement-template slot:** `_engagement-template/01-operator-pack/_examples/TIMELINE-template.md` AND **direct cross-link to Phase 6 deliverable** [`BRAND_GANTT_DISCIPLINE.md`](../../../references/hlk/v3.0/Admin/O5-1/Marketing/Brand/UX%20Designer/canonicals/BRAND_GANTT_DISCIPLINE.md) (P6 will author at the federated home).

**Type-applicability:** ALL engagement types. Every engagement carries a timeline; the per-phase deep-section structure is universally reusable. The Mermaid Gantt SSOT pattern from Phase 6 fits here directly.

**Forward-looking note:** P6 BRAND_GANTT_DISCIPLINE will codify this shape as a brand-governed asset (joint UX-Designer + PMO ownership). This annex's project-timeline pattern becomes the worked example in P6 §6.5.

### 2.5 `competitive-analysis.md` (competitive teardown shape, ~16,000 chars)

**Section structure observed:**
1. Market Overview (size + dynamics + key data points with citations)
2. Competitor Profiles (per-competitor: positioning / pricing / strengths / weaknesses / our differentiator)
3. Competitive Matrix (feature × competitor table)
4. Pricing Comparison
5. Defensible Moats (what we have that they don't)
6. Threat Map (per-competitor: short-term / medium-term threats)
7. Strategic Implications (positioning + pricing + roadmap implications)

**Maps to engagement-template slot:** **NOT YET A TEMPLATE SLOT** — flag for forward-looking inclusion when first competitive-analysis-needed engagement materializes. Reserved at `_engagement-template/01-operator-pack/_examples/COMPETITIVE-ANALYSIS-template.md` (slot defined; content authored when first needed).

**Type-applicability:** Type-3 product engagements where market analysis is required. Less applicable to type-1 customer engagements (where the customer's own market is implicit) or type-2 partner where competitive analysis is a side artifact.

**Forward-looking note:** When I72 RevOps Discovery + Marketing Area Governance ships its competitive-research methodology, this annex's pattern is the prior art it builds on. Probably promotes to `Marketing/Reach/canonicals/` after I72.

### 2.6 `Shopify Plugin Development Plan Annex.md` (annex shape, ~47,500 chars — the largest)

**Section structure observed:**
1. Strategic Architecture and Technical Requirements (front-end + back-end + integration deep-dives)
2. Per-domain technical sections (Front-End / Checkout Extensibility / Back-End / API)
3. Deployment + Release Engineering
4. Compliance + Risk Posture
5. Product Roadmap with Strategic Tiers
6. Operational Continuity
7. Detailed Risk Register
8. Phase-by-phase Investment Schedule

**Maps to engagement-template slot:** `_engagement-template/02-customer-pack/_optional_addons/TECHNICAL-ANNEX-template.md` (NEW reserved slot). The annex shape is for **technical-customer-extensions** — a customer-facing add-on document for engagements where the customer needs deep-architecture transparency beyond the proposal.

**Type-applicability:** Type-2 partner co-built and type-3 product engagements where the customer has technical decision-makers (e.g., SUEZ procurement IT director). Useful when the proposal alone is insufficient for technical sign-off.

**Forward-looking note:** Phase 4.6 HLK_ERP_ARCHITECTURE.md (heavy depth ~900 lines) is in the same shape family as this annex — both are deep-architecture canonicals consumed by technical readers. The Phase 4.6 author should consult this annex's structure for prior art.
