# Supabase (Holistika)

This repo tracks **DDL** under [`migrations/`](migrations/) for the Holistika MasterData / compliance / `holistika_ops` stack. Edge Functions live under [`functions/`](functions/).

The Supabase CLI is pinned as a **`devDependency`** in [`../package.json`](../package.json) (`"supabase": "^2.93.1"`). All invocations below use **`npx supabase`** so every operator and CI runner picks up the pinned version automatically — see [`SOP-HLK_TOOLING_STANDARDS_001.md`](../docs/references/hlk/v3.0/Admin/O5-1/Tech/System%20Owner/SOP-HLK_TOOLING_STANDARDS_001.md) §3.1 for the canonical contract.

## Operator quick card

| Step | Canonical command |
|------|-------------------|
| Authenticate | `npx supabase login` |
| Link repo to project | `npx supabase link --project-ref <PROJECT_REF>` (select Holistika project ref) |
| New DDL change (after `operator-sql-gate` approval) | `npx supabase migration new <short_description>` → edit SQL → PR → `npx supabase db push` (or CI) |
| Inspect remote history | `npx supabase migration list` |
| Reconcile drift | `npx supabase db pull` or `npx supabase migration repair --status <reverted\|applied> <version>` per [Database Migrations](https://supabase.com/docs/guides/deployment/database-migrations) |

> Equivalent form: `npm run supabase -- <subcommand>` (uses the `"supabase": "supabase"` script alias). Either is correct; `npx supabase` is the documented default. **Never bare `supabase`** — that resolves to whatever CLI is globally installed (or fails) and silently drifts from the pinned version.

**Go/no-go:** Do **not** run `db push` while `migration list` shows local-only or remote-only rows — reconcile filenames or repair the ledger first ([`migrations/README.md`](migrations/README.md)).

## Two-plane reminder

- **Migrations:** schema only (`supabase/migrations/*.sql`); applied via `npx supabase db push`.
- **Mirror data (compliance CSV → upserts):** generated by `py scripts/sync_compliance_mirrors_from_csv.py --<register>-only --out artifacts/sql/<file>.sql` → operator-reviewed → applied via Supabase Studio SQL editor **or** `psql -f` against the project connection string. **Not** via `db push` (the migrations folder is for schema, not row data; large mirror DML there inflates the migration ledger).

## Break-glass

DDL through the Dashboard SQL Editor is **only** for emergencies. Afterward: pull or repair migrations so git and `schema_migrations` stay aligned — see [`docs/wip/planning/14-holistika-internal-gtm-mops/reports/operator-sql-gate.md`](../docs/wip/planning/14-holistika-internal-gtm-mops/reports/operator-sql-gate.md).
